# ReelRank — Limitations and Future Work

Honest accounting of what ReelRank does and does not demonstrate (TDD §33). Items are sorted into
three kinds, because they are not the same thing:

- **Limitation of the study** — a bound on how far a measured number generalizes, or a measurement
  caveat. The system works as built; the *claim* has edges.
- **Deliberate non-goal** — explicitly out of scope for v1 by design (TDD §5/§29, decision D16).
  Not a defect and not "unfinished" — a scoping choice.
- **Future work with a designed fix** — a real gap we know how to close, with the fix already
  designed (and in most cases structured so it can be added without an API change).

Each item is traceable to the per-phase history (deviations and known issues recorded at every
phase) in the sibling planning repository's `commit.md`, and to the published results under
[`results/published/`](../results/published/) it concerns.

## Limitations of the study

**Recall at 1M was measured on isotropic-random data, not clustered.** The 1M stretch benchmark
(Phase 11) ran on isotropic random unit vectors, a documented near-worst-case ANN distribution
(high-dimensional distance concentration). Its Recall@10 collapses to 0.19 (M16/ef64) — this is a
**distribution artifact at 10× distractor density and must not be quoted as ReelRank's recall**. The
production-like clustered recall (0.919–0.958 at 100k) was measured; the clustered-1M counterpart
was not run for time-budget reasons. All 1M *latency, throughput, memory, and build-cost* numbers
are valid; only the 1M *recall* number is unrepresentative. (Future work: run the clustered-1M
recall cell — same binary, timing-independent.)

**No LSH baseline.** TDD §7 sketches locality-sensitive hashing as an alternative ANN method, and
vector-db ships an LSH index, but ReelRank benchmarks **exact vs HNSW only**. The
approximate-vs-exact story is complete; the approximate-vs-approximate (HNSW-vs-LSH) comparison was
never built. This is a genuine gap against the strong-portfolio list.

**Oracle regret is in true-affinity units, not TDD §19 reward units.** Simulating counterfactual
oracle interactions in reward units would consume behaviour-RNG draws and break same-seed
determinism (decision D8), so regret is measured on the monotone affinity core of the reward instead
(Phase 4 deviation; documented in `oracle.hpp` and every `summary.json`'s `regret_units_note`). One
consequence must be read carefully: affinity-unit regret **structurally favours similarity-heavy
arms**, so cross-arm regret comparisons between a similarity-only ranker and a feature/diversity
ranker are not apples-to-apples (e.g. the ranker's +2.2% regret in Phase 6 and diversity's +1.2% in
Phase 9 are partly this artifact). Within-arm regret deltas (e.g. Phase 7's learning −21.5%) are
clean.

**Absolute latencies in Phases 4–10 carry concurrent-arm contention.** Those phases ran their
experiment arms concurrently on one machine to save wall time, so every arm's absolute latency is
inflated by cache/memory contention (retrieval p50 drifts ~1.9–4.4 ms across phases for the same
work). Cross-arm comparisons *within* a phase remain like-for-like (identical deterministic feeds
and sampling), and §27's p95<10 ms held in every arm — but **only Phase 11 (serialized, idle
machine) should be quoted for absolute RPS/p99/latency**. Phase 11 exists specifically to be that
clean reference.

**η=0.30 was best only within a 200-interaction horizon.** In the Phase 10 learning-rate sweep the
aggressive long-term rate η=0.30 gave the best overall reward and final alignment, but it also digs
a deeper post-drift trough, and its long-horizon / noisier-regime stability was **not tested**. The
shipped default stays η=0.02; η=0.30 is reported as a finding, not a recommendation.

**1M insert throughput is ~1,563 vec/s.** Building the 1M index took 640 s (M16) / 1,150 s (M32),
versus 2,629–6,931 vec/s at 100k — graph-growth cost. Relevant to any future design that needs
frequent full rebuilds; ReelRank v1 treats reel embeddings as immutable after indexing (decision D2)
and does not rebuild in the hot path, so this does not affect serving latency.

**Synthetic behaviour, no real media.** All users, reels, embeddings, and interactions are
generated by a hand-tuned synthetic model with a *known* ground-truth hidden preference (which is
what makes affinity/regret/alignment measurable at all). The engineering and the trade-off *shapes*
are real; the specific magnitudes are properties of the synthetic model, not of any real platform.
External validity is bounded accordingly — the objective (TDD §34) is to demonstrate the retrieval
and recommendation machinery, not to reproduce a production feed.

**`gaussian()` golden tests use a 1e-12 tolerance, not bit-exactness.** The in-house RNG's integer
and uniform samplers are bit-identical across platforms (exact goldens), but `gaussian()` (Box–Muller
over libm `log`/`cos`) can differ by a few ulps across C libraries, so its regression test allows
1e-12. Determinism holds bit-for-bit on a fixed toolchain; strict cross-libm bit-identity of
gaussian draws is not guaranteed.

**CI is committed but never executed.** The GitHub Actions workflow (macOS + Ubuntu × Debug/Release
+ format check + simulate `--smoke`) has been in the tree since Phase 0 but has **never run** —
the repo has no GitHub remote. The style drift that pre-Phase-5-era files carried (a known
six-file issue, resolved by reformatting in Phase 12) means the tree is now clean under Apple
clang-format 21.0.0 — the locally installed tool — but the CI workflow installs an *unpinned*
clang-format whose decisions may differ. "Builds green, 533/533 tests pass, format clean" is
verified *locally* on this machine, not by CI. Future work: push to a remote so CI actually runs,
and pin the formatter version to one whose behaviour matches the local toolchain.

## Deliberate non-goals (by design, TDD §5/§29, decision D16)

These were scoped out of v1 on purpose. They are listed so the boundary is explicit, not because
they are missing work.

- **No service split / REST / gRPC / distributed deployment.** Single-process simulator only; the
  service-based deployment (TDD §6.3 "later approach") is explicitly deferred past Phase 12.
- **No Kafka / Redis / Kubernetes.** No external infrastructure.
- **No neural rankers.** The second-stage ranker is a transparent feature-weighted linear model
  (chosen partly so feature contributions are inspectable — TDD §14.4).
- **No Thompson sampling / contextual bandits.** Exploration is ε-greedy only.
- **No real media, no frontend.** Synthetic domain only.
- **No vector-db modifications.** vector-db is consumed read-only; all HNSW hardening was delivered
  separately (the `17e434a` rewrite that ReelRank builds on).
- **Reel embeddings are immutable after indexing (D2).** No update/remove on indexed reels in v1 —
  the vector-db update path can create stale entries, so ReelRank never exercises it.
- **`LearningConfig.sessionRate` is intentionally unused.** TDD §11.2 sketches an incremental
  session learning rate, but the implementation follows the TDD §11.3 λ-window session *recompute*
  (no incremental rate) mandated by the Phase 7 plan. The long-term rate η is the only learning rate
  in the system. The field is retained for a possible future incremental variant and flagged here to
  avoid config-surface confusion.

## Future work with a designed fix

These are known gaps where the fix is already designed and, where noted, structured to drop in
without an API change.

**Leaderboard cache for the O(catalog) candidate sources.** The popular / trending / fresh /
exploration / creator-affinity sources scan the whole catalog per request, which profiling clocks at
**~69–72% of request self time** at 100k (HNSW is only ~11–13%)
([`../results/published/phase11/profiling/BOTTLENECKS.md`](../results/published/phase11/profiling/BOTTLENECKS.md)).
This is the cause of both the linear corpus scaling (e2e p99 grows 10.2× for a 10× corpus) and the
thread ceiling (memory-bandwidth bound), **and** of the one §27 nuance: the medium six-source
candidate-generation *stage* crosses 10 ms p95 at T≥8 (though pure HNSW search stays ≤0.84 ms and
e2e p95 stays under the 25 ms target). The designed fix — a deterministic, refreshed-every-K-requests
leaderboard cache — has been recorded since Phase 6 and is structured to be added without an API
change. It is **deliberately deferred**: no §27 target actually *fails*, so per the Phase 11 plan the
bottleneck was profiled and named rather than optimized.

**Fuller feeds under hard diversity caps.** Once online learning concentrates the candidate pool
onto ~2 topics, the hard topic cap binds and delivered feeds shrink from 10.0 to ~6.3 items
(Phase 9). This is documented behaviour of the hard TDD §15.1 rules (no silent relaxation), but it
is the wrong operating point for a real product. The designed knobs are a **topic-aware pool cap**
(diversify the candidate pool *before* the reranker sees it) or a **larger `candidateLimit`** — both
left as future work, both configurable without new interfaces.

**Clustered-1M recall run.** As above — the timing-independent cell simply needs to be executed to
close the strong-portfolio 1M-benchmark item with a representative recall number.

**LSH-vs-HNSW comparison.** vector-db already ships an LSH index; wiring a second `VectorIndex`
adapter over it would let ReelRank report the approximate-vs-approximate trade-off TDD §7 sketches.

## Measurement provenance notes

- Published `metadata.json` files across phases record `reel_rank_dirty: true` (runs predate their
  phase's squash commit) or, for Phase 11, reference merge SHA `537d883` whose tree is byte-identical
  to the squashed Phase 11 commit. This is a history-rewrite bookkeeping artifact, not a
  code-vs-results mismatch; the content mapping is recorded in each phase's `comparison.md`.
- The Phase 5 `exact_vector` run's latency `mean`/`max` are contaminated by a mid-run machine sleep
  (documented in its `comparison.md`); its p50/p95/p99 percentiles over 200k samples are robust and
  are what this project quotes.
